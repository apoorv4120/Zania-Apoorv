CONTEXT
What we were responding to

4 pain points from the brief, and how each one is solved.

The brief criticized the existing chat-dominant interface for four specific reasons. Each is addressed directly in the prototype.

BRIEF 01
Cognitive Overload
The original interface put everything into a single chat panel, all data, all actions, all communication, as one long accordion. As assessments grow, this becomes unmanageable. Users lose track of what's done, what needs attention, and where to look.
↳ OUR SOLUTION
We moved all data out of the single accordion panel into a structured side navigator, where each evidence category is individually visible and clickable, users can jump directly to what they need, minimizing scrolling. The AI assistant, previously the dominant left panel, was repositioned as a collapsible section accessible when needed without occupying permanent screen real estate. The navigator also separates evidence steps (where you see data) from action steps, Vendor Outreach and Response Review, which are visually demarcated as the two places where the analyst actually takes action.
BRIEF 02
High Data Density
A single vendor assessment can generate hundreds of follow-up queries across multiple data sources. Presenting all of this in a flat chat means no scannability at any scale.
↳ OUR SOLUTION
Every data type gets its own purpose-built layout so users always know where to look. Security ratings → 3-column score cards. Trust center documents → expandable list with inline status. Questionnaire → coverage bar chart across 11 control domains with a 76% hero number. Vendor responses → split-pane inbox with a filterable list on the left and a full thread view on the right. The inbox format is what makes Response Review scale: analysts scan the left panel to triage (All 19 / Review 3 / Awaiting 12 / Resolved 4), then click into any item to see the complete communication history and AI analysis on the right, without leaving the page or losing context.
BRIEF 03
Complex Round Histories
Multi-round vendor communication must clearly distinguish what is resolved, what is pending, and what is new. A flat chat collapses this into an unnavigable scroll.
↳ OUR SOLUTION
Response Review is built as a split-pane inbox: a filterable list on the left (All 19 / Review 3 / Awaiting 12 / Resolved 4) and a full thread detail view on the right. Clicking any item opens its complete communication history, every round of outreach and response, alongside the AI's analysis, confidence score, and recommendation. All content is semantically tagged: green = resolved, amber = awaiting vendor, red = needs analyst action. The analyst can Accept, Accept with Note, or Re-request, re-requested items route back into Vendor Outreach as a new bundle with an R2 badge. Round numbers live on each individual item, not the bundle. The 4 live stats, Coverage, Open Gaps, Pending Review, Round, update in real time on every decision.
BRIEF 04
Enterprise Aesthetic
The interface must reflect a modern, clean, enterprise-grade SaaS platform, not a consumer chat tool adapted for compliance use.
↳ OUR SOLUTION
Dark theme, IBM Plex Sans + Mono typography, and a consistent semantic color system signal precision and control throughout. A 3-layer information architecture, global nav shell → assessment pipeline + step navigator → step content, provides spatial hierarchy that scales with complexity. The design is dense without being cluttered: expert compliance teams need all the information visible, not hidden behind progressive disclosure.
Scope coverage

The brief defined 4 steps. All 4 are in the prototype.

The brief restricted scope to the Evidence Collection step, which it described as a 4-step sequence. Here's how each maps to our implementation.

STEP 01
From the brief
"AI agent searches the internet for the vendor's public security documents."
→ OUR IMPLEMENTATION
External Scan · Trust Center · Public Registry
Three AI-executed steps: BitSight/SecurityScorecard/Shodan score cards, a Trust Center document list with inline expand for blocked items, and Public Registry screening (SEC EDGAR, OFAC, adverse media). Each surfaces in its own purpose-built layout.
STEP 02
From the brief
"For missing documents or unanswered questions, gaps are compiled manually or automatically."
→ OUR IMPLEMENTATION
Questionnaire Coverage
142/186 questions pre-filled by AI across 11 control domains. A coverage bar chart shows filled vs. remaining per section. 44 remaining questions surface as a gap item in the Vendor Outreach bundle.
STEP 03
From the brief
"The compiled list of gaps is prepared to be sent to the vendor via email."
→ OUR IMPLEMENTATION
Vendor Outreach
5 gap items compiled into a reviewable bundle. Each item has an AI-drafted request message the analyst can edit or reset. Items are checked off and sent together, not ad-hoc. Re-requested items from Response Review appear here with R2 badges for the next send cycle.
STEP 04
From the brief
"When the vendor responds, AI ingests new info and triggers additional rounds if gaps remain."
→ OUR IMPLEMENTATION
Response Review
An inbox-format split pane: filterable list on the left (All 19 / Review 3 / Awaiting 12 / Resolved 4), full thread detail on the right. Every item shows its complete round history and AI analysis in one place. Analyst accepts, accepts with note, or re-requests, re-requests go back into the Vendor Outreach bundle with an R2 badge. Live stats update on every decision.
ACTION
Visual design, evaluation criterion 2

Every visual choice is a decision, not a default.

The brief explicitly evaluates enterprise visual design. Here's the reasoning behind the four core design decisions.

Dark enterprise theme
Not a stylistic preference, dark reduces visual noise so semantic status signals (green/amber/red) read clearly against a neutral ground. Compliance analysts live in high-stakes workflows; the interface should signal precision and control, not friendliness.
IBM Plex Sans + IBM Plex Mono
IBM Plex pairs a clean humanist sans (UI text, body) with a monospace (data, metrics, timestamps, badges). Monospace communicates precision and machine-origin, right for AI-generated values. This combination is standard in enterprise dev tools and audit software.
Semantic color token system
Every status across all 7 steps maps to exactly one of four tokens, no ad-hoc colors anywhere.
--pos · Complete / Accepted
--ctn · Needs attention / Warning
--crit · Blocked / Critical
--neutral · Sent / Waiting
Density without chaos
Enterprise users (Fortune 500 compliance teams) are expert users, they need information density, not progressive disclosure hiding important signals behind extra clicks. Density is achieved through structure: step-specific layouts, consistent section labels, and tight typographic scale, not by reducing what's shown.
RESULT
AI prototyping skills, evaluation criterion 3

A full design process, run through Claude.

The brief asks how AI was used to bring the UX vision to life. This wasn't code generation, it was a structured design process: research, user testing, iteration, and tooling, all done in collaboration with Claude across multiple working sessions.

01
FRAMEWORK & RESEARCH
Before building anything, we needed to understand the TPRM domain, what compliance analysts actually do, what "hundreds of vendor follow-ups" looks like in practice, and what enterprise-grade means for this type of workflow.
HOW CLAUDE WAS USED
Started by building a structured framework for how to approach the design challenge, problem scope, user goals, constraints, evaluation criteria. Claude then assisted with domain research into TPRM workflows, compliance team structures, and the information architecture patterns used in enterprise GRC tools. This grounded every subsequent design decision in actual domain knowledge rather than assumptions.
02
PERSONA-BASED USER RESEARCH
The brief defines the user but doesn't give us their pain points in detail. We needed first-person perspective from both the design evaluator and the actual end user before making any structural decisions.
HOW CLAUDE WAS USED
Asked Claude to fully embody two distinct personas and respond from their perspective: a UX Director specializing in information architecture and data visualization (to critique the problem space from a design systems lens), and a compliance analyst at a Fortune 500 company (the actual end user, describing day-to-day pain points with vendor assessment workflows). These personas drove the problem definition, particularly the decision to separate data-viewing steps from action-taking steps.
"You are a compliance analyst managing 200+ vendor assessments. Walk me through what breaks down when you use a chat-based evidence collection tool at scale."
03
PROTOTYPE + USABILITY TESTING
Once the first prototype was built, it needed to be tested against real use patterns, not just reviewed aesthetically. We needed to know if the structure actually reduced cognitive load in practice.
HOW CLAUDE WAS USED
Ran multiple rounds of usability testing through Claude using the same two personas, having them interact with the prototype step by step and surface friction points, confusion, and missing context. After each round, findings were compiled into a prioritized fix list and implemented before the next test. This cycle repeated across the full prototype: section by section, interaction by interaction, until each part of the workflow could be navigated without explanation.
04
PROCESS TOOLING & MEMORY
A multi-session, multi-task project like this requires structure, context doesn't persist automatically, accessibility can't be checked manually at every iteration, and working on multiple sections simultaneously requires coordination.
HOW CLAUDE WAS USED
Built dedicated accessibility-checking agents that ran automatically after every round of changes, verifying contrast ratios, ARIA labels, keyboard navigation, and focus states without manual review. Set up Claude's memory system to persist design decisions, component references, and rationale across all sessions so context was never lost between working sessions. Managed multiple terminal sessions simultaneously to work on different parts of the prototype in parallel, one session for layout work, another for interaction logic, another for copy and content. Built skills to standardize how recurring tasks (UX audit, screenshot review, fix-list generation) were executed.
"Before we continue, run the accessibility audit against the current state and give me a P0/P1/P2 fix list."
